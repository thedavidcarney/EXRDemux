## Why this exists

The user's team produces motion graphics for LED screens at concerts. Their pipeline: Blender (Cycles renderer) → multilayer EXR → After Effects, where each lightgroup is animated to a music beat as a separate layer. Typical scenes have 20–30 lightgroups per multilayer EXR plus standard passes.

After Effects has no native way to access individual layers of a multilayer EXR. The team currently uses a third-party AE plugin called **ExtraCtor** to surface each layer one at a time. ExtraCtor works but is **not scriptable** — every layer assignment requires manual UI interaction, blocking the team from automating their post-render AE workflow.

This project: write an open-source replacement that does what ExtraCtor does, but is **scriptable from day one** so it can be driven from `.jsx` files generated by the upstream Blender addon.

## Initial scope

A single AE plugin that:
1. Reads a multilayer EXR file
2. Exposes each layer as something AE can use (importer creating N footage items, vs. effect with a layer-picker — this is an early design decision)
3. Fully scriptable via ExtendScript / future UXP
4. Cross-platform: Windows primary (the team is on Windows), Mac as good-citizen support

Not in scope yet: performance optimization beyond "doesn't suck", color management features beyond passing through what's in the EXR, fancy distribution.

## Hard requirements / decisions already made

- **Must be scriptable.** If the plugin works perfectly but can't be driven from JSX, it fails the requirement.
- **Open source**, MIT or BSD probably.
- **Must handle Blender's EXR conventions**: `OPEN_EXR_MULTILAYER` format, 16-bit half float, DWAB compression, Filmic sRGB colorspace. (These come from the upstream Blender addon — see Reference section.)

## Critical open questions to answer BEFORE designing

1. **Plugin architecture: Importer vs Effect.** AE SDK supports both. ExtraCtor is likely an Effect (you apply it to a layer, parameters pick the channel). An Importer would let AE see a multilayer EXR as N footage items on import — cleaner but possibly more constrained. Read the AE SDK guide and pick.

2. **Legacy C++ SDK vs UXP.** Adobe has been pushing UXP (JavaScript/HTML) for new plugins in Photoshop/InDesign, but AE's UXP support has lagged. Confirm current state before committing to a stack.

3. **Scripting exposure.** For Effects, parameters are scriptable via the layer's `effect()` API. For Importers, scriptability works differently. Need a concrete code example before architecture is locked in.

4. **Build/distribution toolchain.** Windows: Visual Studio + AE SDK headers. Mac: Xcode + AE SDK headers. Adobe SDK requires free developer registration. Output is a `.aex` file users drop in AE's plugin folder.

5. **OpenEXR linking.** OpenEXR is BSD. Static linking avoids users needing to install it separately.

## Suggested first steps

1. Read the AE SDK "Getting Started" + confirm SDK version + recommended plugin type for this use case
2. Find an open-source AE plugin on GitHub to use as scaffolding ("after effects plugin example" / "after effects sdk sample")
3. Build a hello-world plugin from SDK samples; confirm the toolchain works on Windows
4. Then decide importer-vs-effect with eyes open
5. Then write a minimal version that imports a multilayer EXR and exposes ONE layer
6. Then make it scriptable
7. Then handle all layers

## Reference: the upstream Blender addon

The team's existing Blender addon that produces these multilayer EXRs is at:
https://github.com/thedavidcarney/DavidsBlenderProductionToolkit

The addon's `lightgroup_tools/operators.py` contains the compositor setup that writes the EXR — useful as ground truth for what's in the file (lightgroup names, pass names, EXR settings). Don't change that addon from this project.

When this plugin matures, the Blender addon will likely emit a `.jsx` script alongside each render that drives this plugin to auto-set-up the AE comp (naming, blend modes, precomp groupings).
